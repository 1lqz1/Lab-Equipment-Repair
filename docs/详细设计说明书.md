# 基于工作流的高校实验室设备报修管理系统详细设计说明书

## 1. 文档概述

### 1.1 编写目的

本文档用于描述“基于工作流的高校实验室设备报修管理系统”的详细设计方案，明确系统的总体架构、前端设计、后端设计、数据库设计、权限认证设计、核心业务流程、接口设计和日志设计，为后续编码实现、测试验收、项目答辩和维护扩展提供依据。

### 1.2 项目背景

高校实验室设备种类多、使用频繁，传统报修方式主要依赖纸质登记或微信群沟通，容易出现报修记录分散、维修进度不透明、责任划分不清、历史数据难以统计等问题。本系统围绕“报修工单状态流转”组织业务，实现报修、审核、派单、维修、验收、归档全过程管理。

### 1.3 设计目标

- 建立统一的实验室设备报修入口。
- 规范设备报修、管理员审核、维修派单、维修处理、验收归档流程。
- 基于角色控制系统访问权限，区分系统管理员、实验室管理员、维修人员和报修人。
- 保存完整的设备故障、维修处理、验收确认和状态流转记录。
- 支持工单查询、设备查询、实验室管理和基础统计展示。
- 保持前后端结构清晰，便于实训开发、测试和答辩展示。

## 2. 系统总体设计

### 2.1 系统架构

系统采用前后端分离架构：

```text
浏览器客户端
    ↓ HTTP/JSON
Vue 3 + Vite 前端应用
    ↓ Axios 请求
Spring Boot 后端服务
    ↓ MyBatis-Plus
MySQL 数据库
```

### 2.2 技术选型

| 层次 | 技术 | 说明 |
|---|---|---|
| 前端框架 | Vue 3 | 构建单页面应用 |
| 构建工具 | Vite | 前端开发、构建和预览 |
| 路由管理 | Vue Router | 管理登录页、工作台、工单、设备、用户等页面 |
| 状态管理 | Pinia | 保存登录用户、Token 和角色信息 |
| HTTP 请求 | Axios | 与后端 REST 接口通信 |
| 后端框架 | Spring Boot 4.1.0 | 提供后端 Web 服务 |
| 权限认证 | Spring Security | 登录认证、接口授权、关闭默认登录页 |
| 密码加密 | BCrypt | 对 `user.password_hash` 进行加密校验 |
| Token 认证 | JWT | 前后端分离场景下保存登录状态 |
| ORM 框架 | MyBatis-Plus | 简化实体映射、Mapper 和基础 CRUD |
| 数据库 | MySQL | 保存用户、实验室、设备、工单和日志数据 |

### 2.3 仓库结构

当前项目采用一个 Git 仓库统一管理前端、后端和文档：

```text
Lab-Equipment-Repair-fixed
├── backend        后端 Spring Boot 项目
├── frontend       前端 Vue 3 项目
├── docs           数据库脚本、UML 图、设计文档
├── 项目.md        项目开发方案
└── .git           Git 仓库
```

统一仓库管理的好处是前端页面、后端接口、数据库脚本和项目文档可以在同一次提交中保持一致，适合实训项目和答辩交付。

## 3. 用户角色与权限设计

### 3.1 角色定义

系统用户角色由 `user.role` 字段表示。

| 角色编码 | 角色名称 | 主要权限 |
|---|---|---|
| ADMIN | 系统管理员 | 用户管理、实验室管理、设备管理、全部工单管理 |
| LAB_MANAGER | 实验室管理员 | 审核报修、派单、管理实验室设备、查看统计 |
| REPAIRER | 维修人员 | 查看维修任务、开始维修、提交维修结果 |
| REPORTER | 报修人 | 提交报修、查看自己的工单、验收维修结果 |

### 3.2 后端权限控制

后端使用 Spring Security 进行统一认证授权。

主要设计如下：

- 关闭 Spring Security 默认表单登录页。
- 关闭 HTTP Basic。
- 关闭 CSRF，适配前后端分离接口。
- 使用无状态 Session 策略。
- 登录成功后返回 JWT。
- 请求 `/api/**` 接口时通过 JWT 过滤器解析当前用户。
- 根据数据库 `user.role` 字段生成 `ROLE_ADMIN`、`ROLE_LAB_MANAGER`、`ROLE_REPAIRER`、`ROLE_REPORTER` 权限。

权限规则示例：

| 接口范围 | 权限规则 |
|---|---|
| `/api/auth/login` | 允许匿名访问 |
| `/api/admin/**` | 仅 ADMIN |
| `/api/manager/**` | ADMIN、LAB_MANAGER |
| `/api/repair/**` | ADMIN、REPAIRER |
| `/api/**` | 登录用户 |

### 3.3 前端权限控制

前端通过 Pinia 保存登录用户信息和 Token，根据用户角色控制菜单显示和页面访问。

前端权限只用于改善用户体验，真正的权限判断仍以后端 Spring Security 为准。


业务失败示例：

```json
{
  "code": 400,
  "message": "当前工单状态不允许执行该操作",

```

### 5.4 认证登录设计

登录流程：

1. 前端提交用户名和密码到 `/api/auth/login`。
2. 后端通过 `AuthenticationManager` 调用数据库用户认证服务。
3. `DatabaseUserDetailsService` 根据 `username` 查询 `user` 表。
4. 使用 BCrypt 校验明文密码和 `password_hash`。
5. 登录成功后生成 JWT。
6. 前端保存 Token，后续请求携带 `Authorization: Bearer <token>`。
7. 后端 JWT 过滤器解析 Token，构建当前登录用户上下文。

## 6. 核心业务详细设计

### 6.1 工单状态设计

| 状态编码 | 状态名称 | 说明 |
|---|---|---|
| SUBMITTED | 待审核 | 报修人已提交，等待管理员审核 |
| REJECTED | 已退回 | 信息不完整或不符合报修条件 |
| ASSIGNED | 已派单 | 管理员已分配维修人员 |
| IN_PROGRESS | 维修中 | 维修人员已开始处理 |
| REPAIRED | 待验收 | 维修人员已提交维修结果 |
| ACCEPTED | 已验收 | 验收人确认维修完成 |
| CLOSED | 已归档 | 工单流程结束并归档 |
| CANCELLED | 已取消 | 工单取消或作废 |

### 6.2 工单状态流转

```text
SUBMITTED 待审核
    ├── REJECTED 已退回
    └── ASSIGNED 已派单
            ↓
       IN_PROGRESS 维修中
            ↓
       REPAIRED 待验收
            ├── IN_PROGRESS 验收不通过，继续维修
            └── ACCEPTED 已验收
                    ↓
               CLOSED 已归档
```

### 6.3 提交报修

参与角色：报修人、系统管理员。

处理逻辑：

1. 校验当前用户已登录。
2. 根据设备 ID 查询设备信息。
3. 根据设备所属实验室写入工单 `lab_id`。
4. 生成唯一工单编号。
5. 写入 `repair_order`，状态为 `SUBMITTED`。
6. 写入 `order_status_log`，记录“提交报修”。

### 6.4 管理员派单

参与角色：实验室管理员、系统管理员。

处理逻辑：

1. 查询工单。
2. 校验当前工单状态必须为 `SUBMITTED`。
3. 写入维修人员 ID 到 `assigned_to`。
4. 修改工单状态为 `ASSIGNED`。
5. 将设备状态修改为 `REPAIRING`。
6. 写入状态流转日志。

### 6.5 维修人员开始维修

参与角色：维修人员、系统管理员。

处理逻辑：

1. 查询工单。
2. 校验当前状态必须为 `ASSIGNED`。
3. 校验维修人员只能处理分配给自己的任务。
4. 修改状态为 `IN_PROGRESS`。
5. 写入状态流转日志。

### 6.6 完成维修

参与角色：维修人员、系统管理员。

处理逻辑：

1. 查询工单。
2. 校验当前状态必须为 `IN_PROGRESS`。
3. 写入 `maintenance_record`，保存故障原因、处理措施、维修结果、费用和完成时间。
4. 修改工单状态为 `REPAIRED`。
5. 写入状态流转日志。

### 6.7 验收工单

参与角色：报修人、实验室管理员、系统管理员。

处理逻辑：

1. 查询工单。
2. 校验当前状态必须为 `REPAIRED`。
3. 写入 `acceptance_record`。
4. 如果验收通过：
   - 工单状态改为 `ACCEPTED`。
   - 工单自动归档为 `CLOSED`。
   - 设备状态恢复为 `NORMAL`。
5. 如果验收不通过：
   - 工单状态回到 `IN_PROGRESS`。
   - 设备状态保持 `REPAIRING`。
6. 写入状态流转日志。

## 9. 日志设计

### 9.1 系统运行日志

系统运行日志使用 Spring Boot 日志体系，主要用于开发和排错。

记录内容包括：

- 服务启动信息。
- 接口异常信息。
- 登录认证异常。
- 数据库连接或 SQL 执行异常。
- 关键业务失败原因。

### 9.2 业务流转日志

业务流转日志保存到 `order_status_log` 表，是体现系统工作流特征的关键设计。

以下操作必须写入业务日志：

| 操作 | 说明 |
|---|---|
| 提交报修 | 报修人创建工单 |
| 退回报修 | 管理员退回无效或信息不完整工单 |
| 审核派单 | 管理员指派维修人员 |
| 开始维修 | 维修人员开始处理 |
| 完成维修 | 维修人员提交维修结果 |
| 验收通过 | 验收人确认维修完成 |
| 验收不通过 | 工单退回维修中 |
| 工单归档 | 验收通过后关闭流程 |

业务日志用于：

- 工单详情页时间线展示。
- 管理员追踪每个操作人和操作时间。
- 答辩展示系统具备完整工作流追踪能力。
- 后续扩展统计分析和责任追溯。

## 10. 异常处理设计

系统通过全局异常处理器统一处理异常。

| 异常类型 | 返回码 | 说明 |
|---|---|---|
| 参数校验失败 | 400 | 请求参数缺失或格式错误 |
| 业务异常 | 400 | 不符合业务规则 |
| 未登录 | 401 | 未携带 Token 或 Token 无效 |
| 无权限 | 403 | 当前角色不能执行该操作 |
| 系统异常 | 500 | 未预期的服务器错误 |

常见业务异常：

- 用户不存在。
- 密码错误。
- 设备不存在。
- 工单不存在。
- 当前工单状态不允许执行该操作。
- 维修人员只能处理分配给自己的任务。

## 11. 安全设计

### 11.1 密码安全

- 用户密码不保存明文。
- 使用 BCrypt 对密码加密后保存到 `password_hash`。
- 登录时使用 BCrypt 比对明文密码和密文。

### 11.2 Token 安全

- JWT 中保存用户 ID、用户名和角色等必要信息。
- Token 设置过期时间。
- 前端通过请求头携带 Token。
- 后端每次请求都通过过滤器校验 Token。

### 11.3 接口安全

- 所有 `/api/**` 业务接口默认需要登录。
- 管理类接口限制角色访问。
- 前端隐藏无权限菜单，但不能替代后端权限校验。

## 14. 总结

本系统详细设计以“工单状态流转”为核心，将报修人、实验室管理员、维修人员和系统管理员纳入统一工作流中。后端通过 Spring Security、JWT、BCrypt 和 MyBatis-Plus 实现认证授权与业务处理，前端通过 Vue 3、Vue Router、Pinia 和 Axios 实现角色化页面与接口调用。数据库通过工单主表、维修记录表、验收记录表和状态流转日志表保存完整业务过程，能够满足实训项目中需求分析、系统设计、编码实现、测试验收和答辩展示的要求。

# 基于工作流的高校实验室设备报修管理系统项目开发方案

## 1. 项目定位

本项目定位为一个面向高校实验室设备维护场景的工单工作流系统。系统不是简单的设备报修 CRUD，而是围绕“报修单状态流转”组织业务：报修人提交报修，实验室管理员审核并派单，维修人员处理维修任务，报修人或管理员进行验收，最终形成可追踪、可查询、可统计的维修历史记录。

项目建设目标包括：

- 建立统一的实验室设备报修入口。
- 规范报修、审核、派单、维修、验收和归档流程。
- 明确报修人、实验室管理员、维修人员、系统管理员的权限边界。
- 保存设备故障、维修处理、验收确认等历史数据。
- 支撑课程实训中的需求分析、系统设计、编码实现、测试验收和答辩说明。

## 2. 核心业务流程

建议将主流程设计为：

1. 报修人登录系统。
2. 报修人选择设备并填写故障信息。
3. 系统生成待审核报修工单。
4. 实验室管理员审核工单。
5. 管理员将有效工单派发给维修人员。
6. 维修人员接单并填写维修处理结果。
7. 报修人或管理员进行验收。
8. 验收通过后工单关闭归档。
9. 系统保留完整状态流转日志。

工单状态建议：

| 状态编码 | 状态名称 | 说明 |
|---|---|---|
| SUBMITTED | 待审核 | 报修人已提交，等待管理员审核 |
| REJECTED | 已退回 | 信息不完整或不符合报修条件 |
| ASSIGNED | 已派单 | 管理员已分配维修人员 |
| IN_PROGRESS | 维修中 | 维修人员已开始处理 |
| REPAIRED | 待验收 | 维修人员已提交维修结果 |
| ACCEPTED | 已验收 | 报修人或管理员确认维修完成 |
| CLOSED | 已归档 | 工单流程结束并归档 |
| CANCELLED | 已取消 | 工单被取消或作废 |

## 3. 数据库设计建议

### 3.1 用户表 user

用于保存系统用户信息。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| username | VARCHAR(50) | 登录账号 |
| password_hash | VARCHAR(255) | 加密后的密码 |
| real_name | VARCHAR(50) | 真实姓名 |
| phone | VARCHAR(30) | 联系方式 |
| role | VARCHAR(30) | 用户角色 |
| status | VARCHAR(20) | 用户状态 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

角色建议：

- ADMIN：系统管理员
- LAB_MANAGER：实验室管理员
- REPAIRER：维修人员
- REPORTER：报修人

### 3.2 实验室表 lab

用于维护实验室基础信息。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| name | VARCHAR(100) | 实验室名称 |
| location | VARCHAR(150) | 实验室位置 |
| manager_id | BIGINT | 实验室管理员用户 ID |
| description | VARCHAR(255) | 说明 |
| created_at | DATETIME | 创建时间 |

### 3.3 设备表 equipment

用于维护实验室设备档案。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| code | VARCHAR(50) | 设备编号 |
| name | VARCHAR(100) | 设备名称 |
| category | VARCHAR(50) | 设备类别 |
| lab_id | BIGINT | 所属实验室 |
| status | VARCHAR(30) | 设备状态 |
| responsible_user_id | BIGINT | 负责人 |
| purchase_date | DATE | 购置日期 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

设备状态可包括：

- NORMAL：正常
- REPAIRING：维修中
- SCRAPPED：已报废
- DISABLED：停用

### 3.4 报修工单表 repair_order

系统的核心业务表。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| order_no | VARCHAR(50) | 工单编号 |
| equipment_id | BIGINT | 报修设备 |
| reporter_id | BIGINT | 报修人 |
| fault_description | TEXT | 故障描述 |
| urgency | VARCHAR(20) | 紧急程度 |
| location | VARCHAR(150) | 故障地点 |
| contact | VARCHAR(50) | 联系方式 |
| status | VARCHAR(30) | 工单状态 |
| assigned_to | BIGINT | 指派维修人员 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 3.5 维修记录表 maintenance_record

用于保存维修人员的处理结果。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| order_id | BIGINT | 工单 ID |
| repairer_id | BIGINT | 维修人员 ID |
| fault_reason | TEXT | 故障原因 |
| solution | TEXT | 处理措施 |
| result | TEXT | 维修结果 |
| cost | DECIMAL(10,2) | 维修费用，可选 |
| started_at | DATETIME | 开始维修时间 |
| finished_at | DATETIME | 完成维修时间 |

### 3.6 验收记录表 acceptance_record

用于保存验收结果。

| 字段 | 类型建议 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| order_id | BIGINT | 工单 ID |

### 3.7 状态流转日志表 order_status_log

强烈建议保留此表。它是体现“工作流”和“可追踪”的关键。

| 字段 | 类型建议 | 说明 |

| remark | TEXT | 操作备注 |
| created_at | DATETIME | 操作时间 |

## 4. 后端设计建议

后端建议采用 Spring Boot 分层结构。

```text
backend
└── src/main/java/...
    ├── controller
    ├── service
    ├── repository 或 dao
    ├── entity 或 model
    ├── dto
    ├── config
    ├── exception
    └── common
```

### 4.1 Controller 层

负责接收请求、参数校验和返回响应，不建议在 Controller 中写复杂业务逻辑。

主要 Controller：

- AuthController：登录、退出、当前用户信息。
- UserController：用户管理。
- LabController：实验室管理。
- EquipmentController：设备管理。
- RepairOrderController：报修工单管理。
- StatisticsController：基础统计。

### 4.2 Service 层

Service 层负责核心业务逻辑。工单状态流转必须集中在 RepairOrderService 中处理，避免业务规则分散。

RepairOrderService 建议方法：

```java
submitRepairOrder()
approveAndAssign()
rejectOrder()
startRepair()
finishRepair()
acceptRepair()
rejectAcceptance()
closeOrder()
listOrders()
getOrderDetail()
listOrderLogs()
```

每次状态变化应执行：

1. 校验当前用户角色。
2. 校验当前工单状态是否允许该操作。
3. 更新工单状态。
4. 写入状态流转日志。
5. 必要时更新设备状态。

### 4.3 DAO / Repository 层

如果当前项目使用 JDBC，可以先采用简单 DAO 模式：

- UserDao
- LabDao
- EquipmentDao
- RepairOrderDao
- MaintenanceRecordDao
- AcceptanceRecordDao
- OrderStatusLogDao

建议 SQL 文件单独维护，避免 SQL 零散写在业务代码中。

### 4.4 统一响应格式

建议后端返回统一结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

错误示例：

```json
{
  "code": 400,
  "message": "当前工单状态不允许执行该操作",
  "data": null
}
```

### 4.5 REST 接口建议

| 接口 | 方法 | 说明 |
|---|---|---|
| /api/auth/login | POST | 用户登录 |
| /api/auth/me | GET | 获取当前用户 |
| /api/equipment | GET | 查询设备 |
| /api/equipment | POST | 新增设备 |
| /api/repair-orders | POST | 提交报修 |
| /api/repair-orders | GET | 查询工单 |
| /api/repair-orders/{id} | GET | 工单详情 |
| /api/repair-orders/{id}/assign | PUT | 审核派单 |
| /api/repair-orders/{id}/reject | PUT | 退回报修 |
| /api/repair-orders/{id}/start | PUT | 开始维修 |
| /api/repair-orders/{id}/finish | PUT | 完成维修 |
| /api/repair-orders/{id}/accept | PUT | 验收通过 |
| /api/repair-orders/{id}/accept/reject | PUT | 验收不通过 |
| /api/repair-orders/{id}/logs | GET | 查看工单流转记录 |

## 5. 前端设计建议

前端建议围绕角色和工单状态设计页面，不要把所有功能堆在一个后台页面中。

### 5.1 公共页面

- 登录页
- 首页 / 仪表盘
- 个人信息页
- 403 无权限页

### 5.2 报修人页面

- 提交报修
- 我的报修
- 报修详情
- 验收确认

### 5.3 实验室管理员页面

- 待审核工单
- 工单派单
- 全部工单管理
- 设备管理
- 实验室管理

### 5.4 维修人员页面

- 我的维修任务
- 维修处理页面
- 维修记录填写
- 历史维修记录

### 5.5 系统管理员页面

- 用户管理
- 角色管理
- 实验室管理
- 设备管理
- 基础字典管理

### 5.6 前端交互重点

建议重点做好三类交互：

1. 工单状态标签

   使用不同颜色区分待审核、已派单、维修中、待验收、已归档等状态。

2. 工单详情时间线

   在工单详情页展示状态流转日志，让用户清楚看到每一步由谁操作、何时操作、操作结果是什么。

3. 角色化菜单

   登录后根据用户角色展示不同菜单，普通报修人不应看到管理员入口。

## 6. 推荐开发顺序

建议先完成 MVP 主流程，再做扩展功能。

### 6.1 第一阶段：基础框架

- 完成前后端项目启动。
- 完成数据库连接。
- 完成统一响应格式。
- 完成登录接口和登录页面。
- 准备模拟用户数据。

### 6.2 第二阶段：基础数据

- 用户表和用户查询。
- 实验室表和实验室管理。
- 设备表和设备管理。
- 前端设备列表和设备选择组件。

### 6.3 第三阶段：报修主流程

- 报修人提交报修。
- 管理员查看待审核工单。
- 管理员审核派单。
- 维修人员查看任务。
- 维修人员提交维修结果。
- 报修人或管理员验收。
- 工单状态日志记录。

### 6.4 第四阶段：查询和统计

- 按状态筛选工单。
- 按设备、实验室、时间查询工单。
- 首页显示工单数量统计。
- 显示常见故障设备或维修次数统计。

### 6.5 第五阶段：测试和文档

- 补充测试用例。
- 修复主流程缺陷。
- 更新接口文档。
- 更新数据库设计说明。
- 准备答辩演示数据。

## 7. 答辩展示亮点

项目答辩时建议突出以下内容：

- 本项目不是简单报修表，而是基于工作流的设备维修工单系统。
- 使用角色权限区分报修人、管理员、维修人员和系统管理员。
- 使用状态流转日志记录完整处理过程，支持责任追踪。
- 使用设备维修历史数据支撑后续统计分析和维护决策。
- 前后端分离，后端接口清晰，前端页面围绕角色设计。

## 8. 开发注意事项

- 工单状态流转规则必须集中管理。
- 所有重要操作都应写入 order_status_log。
- 前端不要绕过后端权限判断。
- 数据库字段命名、接口字段命名和前端变量命名尽量统一。
- 优先完成主流程，再做统计、通知、附件上传等扩展功能。
- 每完成一个业务节点，就同步更新文档和测试用例。
